org.safs.jvmagent

Class AgentClassLoader

  • java.lang.Object
    • java.lang.ClassLoader
      • org.safs.jvmagent.AgentClassLoader

  • public class AgentClassLoader
extends java.lang.ClassLoader
    Provide a class loader that loads from a custom path. The original intention is to provide a ClassLoader for an Assistive Technologies Java Extension. The new Bootstrap mechanism for Java JVMs does not have access to the system CLASSPATH while extensions are being loaded.

    This classloader doesn't do the security checks normally done by the default ClassLoaders. If path given is null, uses java.class.path.

    Author:
    Carl Nagle 2005.01.06 Original Release, Carl Nagle 2005.01.20 Converted static functions to instance functions., Lei Wang 2013.02.07 Use IndependantLog instead of Log to log message.

    Java Accessibility, Bootstrap, AgentImpl, org.safs.abbot.jvmagent.JVMAgent

    • Constructor Detail

      • AgentClassLoader

        public AgentClassLoader(java.lang.String path)
        Create a class loader that loads classes from the given search paths. The search paths are in typical CLASSPATH format. Though Windows file separators '\' may need to be reversed to prevent character escaping: C:/MyApp.JAR;C:/ChildDir/Another.JAR Unix paths should not have this problem. Using this constructor is the same as using the preferred constructor: AgentClassLoader( path, null);

      • AgentClassLoader

        public AgentClassLoader(java.lang.String path,
                java.lang.ClassLoader parent)
        Preferred constructor creates a class loader that loads classes from the given search paths. Will also refer to any parent ClassLoader as necessary. The search paths are in typical CLASSPATH format. Though Windows file separators '\' may need to be reversed to prevent character escaping: C:/MyApp.JAR;C:/ChildDir/Another.JAR Unix paths should not have this problem.

      • findResourceDirectory

        public static java.io.File findResourceDirectory(java.net.URL url)
        Attempts to locate the system directory where a file or resource was successfully located. The URL is assumed to be a successfully found resource resulting from a call to getResource or getSystemResource. Since these can be found in the file system or in JAR files we attempt to identify the directory where the file or JAR was located. We use the URL from the calling class because this Method may end up using different ClassLoaders and locate resources in different locations than the one needed by the caller. If the URL does NOT ultimately point to a local system directory then this routine returns null.
        Parameters:
        resource - URL to parse for directory information.
        Returns:
        File pointing to containing directory, or null.
